# 日语教学平台技术架构与数据模型草案

## 1. 目标

本草案用于定义一期 MVP 的技术实现边界，服务于以下目标：

- 支撑公众号获客、小程序学习、后台运营三端协同
- 支撑会员、购课、排课、作业、打卡、督学、反馈、活动、续费闭环
- 保证一期可快速上线，同时为后续 AI 能力预留接口

## 2. 总体架构

### 2.1 推荐形态

- 前端一：微信公众号内容页/H5 承接页
- 前端二：微信小程序学员端
- 前端三：Web 教师/运营后台
- 后端：统一业务 API 服务
- 数据库：PostgreSQL
- 对象存储：S3 兼容存储（可用 MinIO，生产可切云厂商对象存储）
- 异步任务：Redis + 队列
- 消息能力：微信订阅消息 / 模板消息

### 2.2 推荐部署方式

- `frontend-admin`：React Web 管理后台
- `frontend-mini`：小程序前端
- `backend-api`：统一 API
- `worker`：异步任务处理
- `postgres`：主数据库
- `redis`：缓存与异步任务队列
- `object-storage`：作业图片、音频、课程素材

### 2.3 为什么这样拆

- 小程序和后台的交互模型完全不同，不建议共用一套前端壳
- 会员、排课、提醒、续费都属于强业务逻辑，应该集中在统一后端
- 提醒、到期通知、批量统计更适合异步任务，避免阻塞主请求

## 3. 技术选型建议

### 3.1 后端

- Python 3.12
- FastAPI
- SQLAlchemy 2.x
- Alembic
- Pydantic
- Celery 或 RQ
- Redis

适合原因：

- 开发速度快
- 数据驱动后台场景适配度高
- 与你现有 `kotobapath` 技术栈接近，团队迁移成本低

### 3.2 管理后台

- React
- Vite
- MUI 或 Ant Design
- TanStack Query
- React Router

适合原因：

- 后台表单、表格、筛选、状态页开发效率高

### 3.3 小程序

- 原生微信小程序 或 Taro

建议：

- 如果团队已有成熟 Web React 团队但小程序经验一般，可用 Taro
- 如果目标是尽快上线且交互不复杂，原生小程序更稳

### 3.4 基础设施

- PostgreSQL：主业务数据
- Redis：缓存、幂等键、消息任务、提醒队列
- MinIO/S3：作业附件、课程封面、活动图片、音频文件

## 4. 系统模块拆分

### 4.1 身份与账号模块

负责：

- 微信登录
- 用户注册/绑定
- 角色识别
- 权限控制

角色：

- 学员
- 教师
- 教务/运营
- 管理员

### 4.2 会员与订单模块

负责：

- 会员方案配置
- 会员开通与续费
- 购课订单
- 支付回调
- 退款状态记录

补充原则：

- 用户可通过单独购买会员或购买课程两种方式获得会员权益
- 用户购买课程后，在课程服务期内自动获得标准会员权益
- 课程教学服务不依赖会员额外续费
- 会员权益内容由后台可配置
- 一期会员权益采用固定权益项配置，不做任意自定义字段

### 4.3 课程与班级模块

负责：

- 课程定义
- 班级定义
- 一对一/小班课区分
- 学员分班
- 课时安排
- 请假与调课记录

补充原则：

- 一对一与小班课共用同一套班级/课次模型
- 一对一视为班级人数上限为 1 的特殊班级
- 排课、调课、补课一期以运营人工处理为主

### 4.4 作业与打卡模块

负责：

- 作业资源管理与发布
- 作业提交
- 每日任务模板配置
- 打卡记录
- 完成状态统计
- 教师点评

补充原则：

- 作业与课程管理绑定，后台维护统一的作业资源库
- 打卡与作业分离，打卡采用每日任务模式
- 每日任务可按课程或班级进行模板配置

### 4.5 督学与消息模块

负责：

- 上课提醒
- 作业逾期提醒
- 打卡未完成提醒
- 会员到期提醒
- 后台待跟进任务

### 4.6 活动模块

负责：

- 活动配置
- 活动报名
- 人数限制
- 收费设置
- 名单管理

补充原则：

- 活动定位以品牌获客入口为主
- 系统支持公开活动、会员活动、免费活动、收费活动
- 活动是否可见、可报名、收费、限会员由后台配置
- 一期支持活动报名取消
- 一期先不做候补名单

### 4.7 学习档案与数据模块

负责：

- 学员基础档案
- 学习目标
- 到课数据
- 作业完成率
- 打卡率
- 教师反馈
- 续费状态

### 4.8 AI 能力预留模块

一期不做重功能，但建议预留：

- AI 配置表
- AI 请求日志
- 跟读/对话结果存储接口
- 学习薄弱项结构

## 6. 核心关系

### 6.1 用户关系

- 一个 `user` 可拥有多个 `role`
- 一个 `user` 可能对应一个 `student_profile`
- 一个 `user` 可能对应一个 `teacher_profile`

### 6.2 课程关系

- 一个 `course` 可开多个 `class`
- 一个 `class` 可有多个 `class_session`
- 一个 `class` 可有多个 `class_enrollment`

### 6.3 学习关系

- 一个 `class_session` 可发布一个或多个 `homework`
- 一个 `homework` 可有多个 `submission`
- 一个 `submission` 可对应多个 `feedback`

### 6.4 运营关系

- 订单驱动会员开通、活动报名或课程购买
- 督学任务由打卡、作业、会员到期等事件触发

## 9. 一期优先实现顺序

### 阶段 1：最小骨架

- 微信登录
- 用户角色模型
- 学员/教师/运营后台基础权限
- 课程、班级、课表基础模型

### 阶段 2：商业闭环

- 会员方案
- 订单与支付
- 入会与续费记录

### 阶段 3：教学闭环

- 作业发布与提交
- 打卡
- 教师反馈
- 学习档案

### 阶段 4：运营闭环

- 活动管理
- 督学任务
- 数据看板

### 阶段 5：AI 预留

- AI 接口网关
- AI 日志表
- AI 功能占位入口

## 10. 一期最容易返工的地方

以下几项必须在开发前确认，否则一定返工：

- 会员和课程是否绑定
- 一对一和小班课的排课逻辑是否一致
- 请假后是否自动补课
- 非会员是否允许直接购课
- 活动是否需要独立支付
- 打卡按日还是按周为主
- 老师反馈是否要求评分结构化

## 10. 推荐结论

如果你准备正式开这个项目，建议：

- 以新仓库重建
- 技术栈可沿用 `kotobapath` 的后端/后台基础能力
- 但数据库模型、API 分组、前端信息架构必须按本项目重做

`kotobapath` 适合复用的是工程脚手架，不适合复用的是业务核心模型。
