# 日语教学平台待商定细节清单

## 1. 文档目的

本清单用于在正式开发前确认关键业务规则，避免数据库、接口、权限和页面结构在开发中途反复返工。

建议原则：

- 先定规则，再建表
- 先定一期边界，再谈扩展
- 先定最小闭环，再补增强功能

## 2. 优先级说明

- `P0`：必须在开发前确认，否则主干设计会返工
- `P1`：建议在开发前确认，否则中后期联调会反复
- `P2`：可以先保留默认方案，后续迭代再细化

## 3. 商业模式

### 3.1 会员的本质是什么

- 优先级：`P0`
- 待确认：
  - 会员是“身份权益”还是“直接包含课程/课时”
  - 会员是否只是享受会员价、活动资格、督学服务
  - 会员是否与购课完全独立
- 为什么要先定：
  - 直接影响订单结构、续费逻辑、权限判断和页面文案

### 3.2 非会员是否可以直接购课

- 优先级：`P0`
- 待确认：
  - 非会员能否购买一对一课程
  - 非会员能否购买小班课程
  - 非会员能否报名活动
- 为什么要先定：
  - 直接影响支付入口、产品定价、报名限制和转化路径

### 3.3 一期有哪些收费对象

- 优先级：`P0`
- 待确认：
  - 收费项目是否包含会员、课程、活动
  - 是否还存在试听费、材料费、报名费
- 为什么要先定：
  - 直接影响商品模型、订单表和后台管理项

### 3.4 是否支持退款 kyc

- 优先级：`P1`
- 待确认：
  - 会员是否允许退款
  - 课程开课后是否允许退款
  - 活动报名后是否允许退款
  - 退款是全额还是按规则扣费
- 为什么要先定：
  - 影响订单状态、支付回调、人工审核流程

## 4. 课程与教学形态

### 4.1 一期上线哪些课程类型

- 优先级：`P0`
- 待确认：
  - 是否只做 0 起步和初级
  - 是否同时做一对一和小班
  - 是否支持试听课
- 为什么要先定：
  - 影响课程表结构、班级模型和销售入口

### 4.2 课程与班级的关系

- 优先级：`P0`
- 待确认：
  - 一个课程是否可开多个班
  - 一对一是否也按“班级”建模
  - 课次是否固定绑定班级
- 为什么要先定：
  - 影响排课模型、课表模型、教师绑定关系

### 4.3 课时消耗规则

- 优先级：`P1`
- 待确认：
  - 课程是否按课次计费
  - 缺课是否扣课时
  - 补课是否单独记录
- 为什么要先定：
  - 影响订单、课时包、出勤和补课逻辑

## 5. 排课、请假、补课

### 5.1 排课由谁主导

- 优先级：`P0`
- 待确认：
  - 学员能否自主约课
  - 是否全部由教务/运营人工排课
  - 教师是否可自行调整课表
- 为什么要先定：
  - 影响后台页面复杂度和流程设计

### 5.2 请假规则

- 优先级：`P0`
- 待确认：
  - 是否允许线上请假
  - 请假截止到开课前多久
  - 超时未请假按什么处理
- 为什么要先定：
  - 影响课表状态、缺席记录和补课资格

### 5.3 补课与调课规则

- 优先级：`P0`
- 待确认：
  - 是否允许补课
  - 调课是否只能由运营处理
  - 补课是否有有效期
  - 同一个课次是否可多次调整
- 为什么要先定：
  - 影响排课数据结构和运营工作流

## 6. 作业、打卡、反馈

### 6.1 作业机制

- 优先级：`P0`
- 待确认：
  - 是否每节课默认布置作业
  - 作业支持哪些提交形式：文字、图片、音频
  - 是否设置截止时间
- 为什么要先定：
  - 影响作业表结构、上传能力和提醒逻辑

### 6.2 打卡机制

- 优先级：`P0`
- 待确认：
  - 按日打卡还是按周打卡
  - 是否允许补打卡
  - 打卡是否必须带文字
  - 是否支持图片
- 为什么要先定：
  - 影响打卡任务生成逻辑和统计口径

### 6.3 老师反馈形式

- 优先级：`P0`
- 待确认：
  - 反馈是纯文本还是文本 + 评分 + 标签
  - 是否有固定模板
  - 是否记录阶段性学习建议
- 为什么要先定：
  - 影响教师工作台和学习档案结构

### 6.4 督学提醒规则

- 优先级：`P1`
- 待确认：
  - 作业逾期多久提醒
  - 打卡未完成何时提醒
  - 提醒几次
  - 提醒失败是否进入人工跟进
- 为什么要先定：
  - 影响异步任务、通知策略和运营工作量

## 7. 活动与留存

### 7.1 活动定位

- 优先级：`P0`
- 待确认：
  - 活动是会员权益还是独立业务
  - 活动是否公开报名
  - 活动是否收费
- 为什么要先定：
  - 影响报名限制、商品模型和活动页面设计

### 7.2 活动参与规则

- 优先级：`P1`
- 待确认：
  - 是否限制人数
  - 是否支持候补名单
  - 报名后能否取消
- 为什么要先定：
  - 影响活动状态流和名单管理

## 8. 用户、角色、权限

### 8.1 一期有哪些角色

- 优先级：`P0`
- 待确认：
  - 是否固定为学员、教师、教务/运营、管理员
  - 是否存在顾问/销售角色
  - 一个用户是否可拥有多个角色
- 为什么要先定：
  - 影响权限模型和后台菜单结构

### 8.2 教师端和运营端是否共用后台

- 优先级：`P0`
- 待确认：
  - 是一个 Web 后台多角色权限，还是拆成两个后台
- 为什么要先定：
  - 影响前端工程结构和登录入口设计

### 8.3 学员端是否只用小程序

- 优先级：`P1`
- 待确认：
  - 是否需要 H5 备用入口
  - 公众号是否只做内容承接
- 为什么要先定：
  - 影响前端端形态和登录方式

## 9. 公众号、小程序、后台边界

### 9.1 公众号承担什么职责

- 优先级：`P0`
- 待确认：
  - 公众号是否只负责内容展示和导流
  - 是否承接咨询表单
  - 是否承接活动宣传
- 为什么要先定：
  - 影响页面范围和内容管理需求

### 9.2 小程序承担什么职责

- 优先级：`P0`
- 待确认：
  - 是否包括会员开通、支付、课表、作业、打卡、活动报名、学习档案
- 为什么要先定：
  - 影响学员端页面清单和开发量评估

### 9.3 Web 后台承担什么职责

- 优先级：`P0`
- 待确认：
  - 是否统一承载课程、班级、排课、作业、订单、活动、数据
- 为什么要先定：
  - 影响后台信息架构和权限模型

## 10. 支付与订单

- 优先级：`P1`
- 待确认：
  - 是否只接微信支付

### 10.2 订单状态流

- 为什么要先定：

## 11. 内容与配置

- 优先级：`P1`

- 公众号、小程序、后台各自负责什么

## 16. 已确认结论（第一轮）

### 16.1 会员与课程关系

- 会员权益可通过两种方式获得：
  - 单独购买会员
  - 购买课程后自动获得
- 单独购买会员时，按月度或季度等时间有效期享受会员权益
- 用户购买课程后，在课程服务期内自动享受会员权益
- 课程本身的教学服务不依赖会员单独续费
- 会员权益内容由后台可配置、可编辑
- 一期默认课程赠送的是标准会员权益，不拆分会员层级

### 16.2 非会员购课规则

- 非会员允许直接购买课程
- 用户购买课程后，自动获得该课程服务期内的会员权益
- 不要求用户必须先单独开通会员再购课

### 16.3 课程形态

- 一期同时支持一对一和小班课
- 系统统一采用同一套班级/课次/报名模型
- 一对一视为人数上限为 1 的特殊班级
- 小班课视为人数上限大于 1 的普通班级
- 一期不做复杂自动排课，先由运营人工管理

### 16.4 排课、请假、补课

- 排课由运营/教务人工处理
- 学员可线上提交请假申请
- 请假截止时间支持后台配置，默认课前 24 小时
- 超时未请假默认记为缺席
- 是否补课由运营人工判定
- 补课和调课全部由运营后台人工处理
- 系统一期只负责记录、提醒、留痕，不做自动改排

### 16.5 打卡规则

- 打卡采用每日任务模式
- 打卡与作业分离
- 每日任务可为固定学习任务，如单词、题目、跟读等
- 后台可按课程或班级配置每日任务模板
- 系统按任务完成度判断是否完成打卡
- 支持连续打卡统计
- 支持有限补打卡机制

### 16.6 作业与老师反馈

- 作业纳入课程管理体系，后台统一管理
- 后台维护统一的作业资源库
- 作业资源可按课程统一配置，并分配到具体班级、课次或学员
- 作业提交支持文字、图片、音频
- 教师可进行文本点评
- 评分和标签一期先作为可选预留
- 作业反馈与阶段性反馈进入学习档案

### 16.7 活动定位

- 活动定位以品牌获客入口为主，不做过死限制
- 系统支持公开活动、会员活动、免费活动、收费活动
- 活动是否可见、是否可报名、是否收费、是否限会员，由后台配置
- 活动报名与课程报名分开管理
- 一期活动重点用于引流、品牌展示、社群活跃和转化辅助

### 16.8 角色与后台边界

- 教师、教务/运营、管理员共用一个 Web 后台
- 后台通过角色权限划分菜单与操作范围
- 一期角色先定为：
  - `student`
  - `teacher`
  - `operator`
  - `admin`
- 一期不单独设置销售/顾问角色
- 若后续需要销售职能，先并入 `operator`

### 16.9 三端职责边界

- 公众号负责品牌展示、课程介绍、教师介绍、活动宣传、内容输出、咨询承接、跳转小程序
- 小程序负责用户登录、会员开通、课程购买、课表查看、作业提交、每日任务打卡、活动报名、学习档案查看、消息提醒接收
- Web 后台负责课程管理、班级/课次管理、学员管理、会员管理、订单管理、作业资源管理、作业点评与反馈、活动管理、数据报表、运营跟进
- 一期咨询入口主要放在公众号侧承接
- 后台统一管理咨询线索

### 16.10 支付与订单

- 一期支付方式先支持微信支付
- 商品类型至少包括会员、课程、活动
- 订单状态至少包括：
  - `pending`
  - `paid`
  - `cancelled`
  - `refund_pending`
  - `refunded`
- 支付回调负责更新订单状态
- 后台保留人工调整/补单能力作为兜底
- 一期预留退款状态管理，初期可人工处理退款流程

### 16.11 会员权益具体可配置项

- 会员权益一期采用固定权益项配置，不做任意自定义字段
- 通用会员权益包括：
  - 基础课程访问权
  - 会员价购买资格
  - 活动报名资格
  - 活动会员价
  - 每日任务/督学服务
  - 消息提醒服务
- 作业提交、教师反馈、学习档案查看优先归属课程服务权益，不作为纯会员专属能力
- 后台可对每个会员方案配置：
  - 是否启用某项权益
  - 有效期
  - 备注说明
- 后台支持对单个用户手工调整或补充权益
- 基础课程访问权一期定义为：
  - 可访问基础课程内容/资源
  - 可参加基础课程班级
  - 基础班级仍由运营分配，不支持用户自行无限加入

### 16.12 基础课程与精品课程边界

- 基础课程偏会员内容
- 精品课程偏独立售卖课程
- 基础课程主要面向 0 起步、初级的标准化学习内容
- 基础课程可作为会员权益的一部分提供
- 精品课程具有明确服务周期、班级、教师、课次和教学交付
- 精品课程购买后，自动获得课程服务期内的会员权益

### 16.13 每日任务模板

- 一期每日任务仅支持两类：
  - 词汇任务
  - 题目练习
- 每日任务由系统自动生成
- 一期不做复杂个性化推荐
- 词汇任务默认规则为每日随机 5 个单词
- 题目练习默认规则为每日随机 3 个语法题
- 每日任务完成后视为当日打卡完成
- 后台可配置任务类型、题量、适用课程或班级
- 每日任务优先从课程或班级绑定题库中随机抽取
- 如果用户当前没有绑定课程，或课程下没有可用题目，则从平台总题库中抽取

### 16.14 退款规则

- 会员、课程、活动均支持退款状态管理
- 退款流程一期采用人工审核与人工判定
- 系统记录退款申请、退款状态、退款原因、处理人、处理时间
- 一期先不做自动退款金额计算
- 退款后的会员权益或课程权益由后台人工确认回收
- 课程开始后是否允许退款，一期也采用人工判定
- 订单退款后保留完整操作日志

### 16.15 一期报表核心指标

- 一期报表至少展示以下指标：
  - 咨询人数
  - 课程购买人数
  - 会员开通人数
  - 活动报名人数
  - 在读学员数
  - 到课率
  - 作业完成率
  - 每日任务完成率
  - 续费率
  - 订单金额
  - 退款人数
- 报表展示形态至少包括：
  - 总览看板
  - 学员明细页
- 学员明细页建议展示：
  - 所属课程/班级
  - 会员状态
  - 到课情况
  - 作业完成情况
  - 每日任务完成情况
  - 续费状态
  - 跟进记录
- 一期“咨询人数”口径先定义为公众号咨询表单提交人数

### 16.16 活动取消与候补规则

- 一期支持活动报名取消
- 一期先不做候补名单
- 名额满了后停止报名
- 活动取消报名后的退款处理继续采用人工判定

### 16.17 作业资源库录入方式

- 一期作业资源库支持后台手工录入
- 批量导入能力先预留，不作为一期必须功能

### 16.18 企业微信与社群能力

- 一期不接入企业微信
- 一期不做微信群自动化或社群系统集成
- 社群运营相关工作先走人工流程
- 系统不承担群运营自动化能力
