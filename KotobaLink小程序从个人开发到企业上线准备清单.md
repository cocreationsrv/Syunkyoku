# KotobaLink 小程序上线准备清单

## 1. 文档目的

本清单用于说明 KotobaLink 从当前个人开发和 Demo 阶段，推进到企业主体正式上线微信小程序时，需要重点处理的四块事项：

- 备案
- 配置
- 审核
- 微信支付商户号

这份文档不展开技术开发排期，只聚焦正式上线前必须补齐的手续与配置。

## 2. 上线前必须准备的手续

| 事项 | 是否必须 | 说明 |
| --- | --- | --- |
| 企业或个体主体小程序账号 | 必须 | 商业项目不建议长期使用个人主体上线 |
| 微信认证 | 必须 | 很多小程序能力和支付依赖认证 |
| 小程序备案 | 必须 | 未备案不能正常发布上线 |
| 服务类目设置 | 必须 | 类目必须和教学/服务实际业务一致 |
| 管理员 / 开发者 / 体验者配置 | 必须 | 用于协作开发、真机调试和提审 |
| 隐私政策 / 用户协议 / 收费说明 | 必须 | 审核和合规都需要 |
| 微信支付商户号 | 若收费则必须 | KotobaLink 有会员、课程、活动收费，基本需要 |

## 3. 需要提前准备的资料

### 3.1 主体资料

- 企业营业执照或个体工商户营业执照
- 法人 / 经营者身份证信息
- 联系人手机号
- 联系邮箱

### 3.2 小程序资料

- 小程序名称
- 小程序 Logo
- 小程序简介
- 服务类目
- 管理员微信号

### 3.3 合规资料

- 用户协议
- 隐私政策
- 收费说明
- 退款规则
- 联系方式

### 3.4 支付资料

- 微信支付商户号
- 结算账户
- 商户联系人
- 商户平台证书 / API 密钥
- 小程序 AppID 与商户号绑定信息

## 4. 小程序备案详细说明

| 事项 | 为什么要做 | 需要准备什么 |
| --- | --- | --- |
| 小程序备案 | 未备案不能正常发布上线 | 主体信息、负责人信息、服务内容说明、服务器信息 |
| 备案信息一致性 | 备案、主体、类目、实际功能必须一致 | 课程、会员、活动、作业、督学等业务描述要与实际页面一致 |
| 备案后变更管理 | 后续主体或服务范围变化可能影响持续运营 | 名称、主体、服务范围变化时要重新核对备案要求 |

### 4.1 对 KotobaLink 的特殊意义

KotobaLink 不是纯展示型小程序，涉及：

- 课程服务
- 会员权益
- 活动报名
- 作业上传
- 学习档案
- 提醒服务

所以备案信息和实际业务说明不能写得过于模糊。

## 5. 配置项详细说明

| 配置项 | 作用 | 上线前要求 |
| --- | --- | --- |
| 正式 AppID | 小程序正式身份标识 | 必须切换到企业主体的正式 AppID |
| request 合法域名 | 允许小程序访问后端接口 | 正式后端域名需配置到后台白名单 |
| upload 合法域名 | 承接作业音频、图片等上传 | 上传服务或对象存储域名需提前配置 |
| 业务域名 | 承接协议页、说明页等网页内容 | 若有 H5 页面，也要加入业务域名 |
| 消息与回调配置 | 承接任务提醒、支付通知等 | 正式环境需使用生产回调地址和正式配置 |
| 开发者 / 体验者配置 | 支持协作和真机测试 | 上线前需要配置测试人员、教务和运营体验账号 |

### 5.1 对 KotobaLink 的实际影响

如果这些配置没准备好，会直接影响：

- 小程序无法正常请求后端
- 作业音频/图片无法上传
- 支付回调无法落单
- 提醒能力无法接通
- 提审时功能无法完整演示

## 6. 审核详细说明

### 6.1 审核会看什么

- 主体和类目是否匹配
- 页面功能和实际业务是否一致
- 收费链路是否清晰
- 隐私政策和用户协议是否齐全
- 用户数据收集是否说明清楚

### 6.2 KotobaLink 容易被重点关注的内容

- 课程收费
- 会员权益
- 活动报名
- 作业上传
- 学习档案
- 消息提醒
- 退款说明

### 6.3 审核前建议准备

- 测试账号
- 测试路径说明
- 用户协议
- 隐私政策
- 收费说明
- 退款规则
- 客服联系方式

### 6.4 常见风险

- 类目不匹配
- 收费说明不清
- 隐私收集说明不完整
- 页面承诺的服务和实际功能对不上

## 7. 微信支付商户号详细说明

| 事项 | 说明 | 对 KotobaLink 的意义 |
| --- | --- | --- |
| 为什么需要商户号 | 小程序内正式收费要通过微信支付商户体系收款 | 会员、课程、活动三类商品都依赖正式收款能力 |
| 开通前提 | 通常要求主体已认证，并具备可结算账户 | 没有商户号就只能停留在 Demo 或 mock 支付阶段 |
| 核心准备资料 | 主体信息、结算账户、联系人、商户资料、证书和密钥 | 后端要接支付回调、签名和订单状态流转 |
| AppID 绑定 | 正式小程序 AppID 需要与正式商户号绑定 | 购课、会员开通、活动报名才能在正式环境支付 |
| 上线前联调 | 必须验证下单、支付成功、回调、退款状态更新 | 否则课程权益、会员权益和订单状态会对不上 |
| 退款处理 | 当前方案是人工审核退款，但系统仍要记录退款状态 | 商户号和后台要支持退款申请、人工判定和状态留痕 |

## 8. 一句话总结

从个人开发到企业正式上线，最关键的不是继续补 Demo，而是把下面四块补齐：

- 备案
- 配置
- 审核
- 支付商户号

这四块处理好，KotobaLink 才能从“能演示”走到“能上线”。
