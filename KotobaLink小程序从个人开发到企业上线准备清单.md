# KotobaLink 小程序上线准备清单

## 1. 文档目的

本清单以 `大陆企业主体正式上线微信小程序` 为主线，整理 KotobaLink 从当前 Demo 阶段走到正式上线前需要准备的材料和事项。

如果客户主体是 `日本公司`，文中会在关键节点补充 `日企备注`，提示哪些地方不能直接照搬大陆企业流程。

本文重点聚焦：

- 主体与认证
- 备案
- 配置
- 审核
- 微信支付商户号与支付关系

## 2. 上线前必须准备的事项

| 事项 | 是否必须 | 说明 |
| --- | --- | --- |
| 企业主体小程序账号 | 必须 | 正式商业项目建议直接使用企业主体 |
| 微信认证 | 必须 | 很多正式能力和支付依赖认证 |
| 小程序备案 | 必须 | 未备案通常不能正式发布上线 |
| 服务类目设置 | 必须 | 类目必须和教学/服务实际业务一致 |
| 管理员 / 开发者 / 体验者配置 | 必须 | 用于协作开发、真机调试和提审 |
| 隐私政策 / 用户协议 / 收费说明 / 退款规则 | 必须 | 审核和合规都需要 |
| 微信支付商户号 | 若收费则必须 | KotobaLink 的会员、课程、活动收费都依赖它 |

### 日企备注

- 如果主体是日本公司，主体注册与认证通常走 `境外主体 / 海外主体` 路径。
- 不能默认所有步骤和大陆企业完全一样，尤其是：
  - 备案要求
  - 支付商户方案

## 3. 需要提前准备的材料

### 3.1 主体材料

- 营业执照
- 法人身份证信息
- 联系人手机号
- 联系邮箱
- 管理员微信号

### 3.2 小程序材料

- 小程序名称
- 小程序 Logo
- 小程序简介
- 服务类目

### 3.3 合规材料

- 用户协议
- 隐私政策
- 收费说明
- 退款规则
- 客服/联系方式

### 3.3.1 用户协议与隐私政策模板说明

- 微信平台没有统一官方的 `用户协议模板` 可直接套用。
- 隐私政策这块，微信更强调：
  - 你是否按实际业务填写了 `用户隐私保护指引`
  - 页面里是否能提供对应的 `隐私政策` 和 `用户协议`
- 实际项目里，通常做法是：
  - 先用模板或生成器出一版初稿
  - 再按 KotobaLink 的真实业务补充和修改
  - 最后再做法务/业务校对

### 3.4 支付材料

- 微信支付商户号
- 结算账户
- 商户联系人
- 商户平台 API 密钥 / 证书
- 小程序 AppID 与商户号绑定信息

### 日企备注

按当前公开资料，若主体是日本公司，常见还需要额外准备：

- 日本公司主体登记资料
- 联系人身份证件
- 申请公函
- 运营授权书
- 近三个月手机或银行对账单

正式提交前仍应以微信平台后台当时要求为准。

## 4. 当前平台要求里最需要注意的点

### 4.1 主体与认证

- 企业主体可在微信公众平台注册小程序。
- 企业主体可通过主体认证获得更多正式能力。
- 已认证小程序通常才能进一步申请或接通正式支付。

### 4.2 类目

- 非个人主体、个人主体、海外主体开放的服务类目不同。
- KotobaLink 属于教学/服务型业务，类目、页面功能、收费说明必须一致。

### 4.3 开发者与体验者

按当前公开资料，已认证小程序通常可绑定更多开发者与体验者。常见整理口径是：

- 未认证组织类小程序：最多 10 个开发者、20 个体验者
- 已认证小程序：最多 20 个开发者、40 个体验者

这块适合你当前 Demo 阶段就先预留账号协作方式。

### 日企备注

- 如果是日本公司主体，主体注册国家/地区、类目范围、认证材料要求都可能和大陆企业不同。
- 所以不能直接拿大陆企业后台里看到的类目和认证步骤做最终判断。

## 5. 备案详细说明

| 项目 | 说明 |
| --- | --- |
| 是否需要备案 | 大陆企业主体正式上线通常需要备案 |
| 备案重点 | 主体信息、负责人信息、服务内容说明、服务器信息 |
| 一致性要求 | 备案、主体、类目、实际页面功能要一致 |
| 对 KotobaLink 的要求 | 课程、会员、活动、作业上传、学习档案、提醒服务等业务描述要和实际页面一致 |

### KotobaLink 为什么要特别注意备案

KotobaLink 不是单纯展示型小程序，而是包含：

- 课程服务
- 会员权益
- 活动报名
- 作业上传
- 学习档案
- 提醒服务

所以备案信息和业务描述不能写得过于模糊。

### 日企备注

- 如果主体是日本公司，`备案` 这一块不能默认完全按大陆企业同流程处理。
- 正式上线前，必须按境外主体口径单独确认当前要求。

## 6. 配置项详细说明

| 配置项 | 作用 | 上线前要求 |
| --- | --- | --- |
| 正式 AppID | 小程序正式身份标识 | 必须切换到正式主体的 AppID |
| request 合法域名 | 允许小程序访问后端接口 | 正式后端域名需加入后台白名单 |
| upload 合法域名 | 承接作业音频、图片等上传 | 上传服务或对象存储域名需提前配置 |
| 业务域名 | 承接协议页、说明页等网页内容 | 若有 H5 页面，也要加入业务域名 |
| 消息与支付回调配置 | 承接任务提醒、支付通知等 | 正式环境需使用生产回调地址和正式配置 |
| 开发者 / 体验者配置 | 支持协作和真机测试 | 上线前需要配置测试人员、教务和运营体验账号 |

### 配置项对 KotobaLink 的实际影响

如果这些配置没准备好，会直接导致：

- 小程序无法正常请求后端
- 作业音频/图片无法上传
- 支付回调无法落单
- 提醒能力无法接通
- 提审时功能无法完整演示

### 日企备注

- 如果主体是日本公司，技术配置本身大体类似，但需要额外确认：
  - 最终 AppID 对应的主体
  - 支付方案对应的回调与商户绑定方式

## 7. 审核详细说明

审核通常会重点看：

- 主体与类目是否匹配
- 页面功能与实际业务是否一致
- 收费链路是否清晰
- 隐私政策、用户协议、退款说明是否完整
- 用户数据收集是否说明清楚

### KotobaLink 容易被重点关注的内容

- 课程收费
- 会员权益
- 活动报名
- 作业上传
- 学习档案
- 消息提醒
- 退款说明

### 审核前建议准备

- 测试账号
- 测试路径说明
- 用户协议
- 隐私政策
- 收费说明
- 退款规则
- 客服联系方式

### 常见风险

- 类目不匹配
- 收费说明不清
- 隐私收集说明不完整
- 页面承诺的服务与实际功能对不上

### 日企备注

- 如果主体是日本公司，还要额外注意：
  - 境外主体资料是否准备齐全
  - 业务描述是否和主体、类目、支付方案一致

## 8. 微信支付商户号详细说明

| 项目 | 说明 | 对 KotobaLink 的意义 |
| --- | --- | --- |
| 为什么需要商户号 | 小程序内正式收费要通过微信支付商户体系收款 | 会员、课程、活动三类商品都依赖正式收款能力 |
| 开通前提 | 通常要求主体已认证，并具备可结算账户 | 没有商户号就只能停留在 Demo 或 mock 支付阶段 |
| 核心准备资料 | 主体信息、结算账户、联系人、商户资料、证书和密钥 | 后端要接支付回调、签名和订单状态流转 |
| AppID 绑定 | 正式小程序 AppID 需要与正式商户号绑定 | 购课、会员开通、活动报名才能在正式环境支付 |
| 上线前联调 | 必须验证下单、支付成功、回调、退款状态更新 | 否则课程权益、会员权益和订单状态会对不上 |
| 退款处理 | 当前方案是人工审核退款，但系统仍要记录退款状态 | 商户号和后台要支持退款申请、人工判定和状态留痕 |

## 9. 支付关系说明

KotobaLink 正式收费时，最需要看清的是下面这组关系：

- `小程序 AppID`
  - 代表正式上线的小程序身份
- `小程序主体`
  - 决定认证、类目、备案、支付路径
- `微信支付商户号`
  - 决定资金结算主体
- `商户平台 API 密钥 / 证书`
  - 用于支付签名、回调验签、退款等服务端能力
- `支付回调地址`
  - 后端接收支付成功通知，更新订单和权益状态

实务上通常是：

1. 小程序先完成主体认证
2. 商户号准备好
3. 商户号与小程序 AppID 进行授权绑定
4. 后端配置商户号、密钥、回调
5. 联调支付成功、退款、订单状态和权益发放

如果这些关系没理清，会直接导致：

- 支付成功但订单不更新
- 课程权益和会员权益不同步
- 退款后状态无法回收
- 小程序能下单但不能正式收款

### 日企备注

如果主体是日本公司，这里最需要先确认的是：

- 是否能直接使用当前主体接商户号
- 是否需要走跨境支付或服务商方案
- 小程序主体和收款主体是否一致

如果这里没先确认，最容易出现的问题是：

- Demo 看起来能卖课，但正式版无法收款
- 小程序主体和收款主体不一致，导致流程卡在支付接入
- 订单、课程权益、会员权益和退款状态的归属不清

## 10. 一句话总结

对 KotobaLink 来说，正式上线的主线应先按 `大陆企业主体` 材料和流程准备；  
如果客户主体是 `日本公司`，重点不是重写整套流程，而是在 `主体认证`、`备案`、`支付方案` 这三个地方做额外确认。
